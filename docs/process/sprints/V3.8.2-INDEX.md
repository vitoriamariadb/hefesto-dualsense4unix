# Wave V3.8.2 — Boot saudável: destrava o ciclo daemon-em-D-state + GUI imkillable + notificações de boot

Índice da wave V3.8.2. Motivação: depois da v3.8.1 a mantenedora reportou três sintomas
combinados que faziam o boot do PC ser frustrante — "ao ligar o PC ele fica falando que tem
algo não instalado", "o app travou legal, não consigo nem dar um kill nele", "o controle fica
conectando e desconectando, o daemon dá pau legal".

A sessão diagnosticou as três coisas como bugs independentes que se reforçavam, corrigiu cada
uma com fixes cirúrgicos e validou runtime na máquina — daemon novo sobe sem `EBADF`, GUI
morre em <3s no SIGTERM (mesmo com diálogo modal), notify spam silenciado.

## Sprints

| ID | Sprint | Entrega |
|----|--------|---------|
| 145 | BUG-DAEMON-BOOT-DSTATE-LOOP-01 | (a) `os.close(fd)` duplicado em `single_instance.py` virava `OSError EBADF` → daemon morria no boot → systemd `Restart=on-failure` ciclava (era o "loop conecta/desconecta"). Cleanup centralizado com `contextlib.suppress(OSError)`. (b) `pydualsense.init()` entrava em `D (disk sleep)` em hidraw degenerado e ficava imkillable. Agora roda em `threading.Thread(daemon=True)` com timeout 5s — marca offline se passar |
| 146 | BUG-GUI-IMKILLABLE-01 | (a) Signal handler defensivo com 3 camadas (`Gtk.main_quit()` direto + `idle_add(quit_app)` + watchdog 2s → `os._exit(128+sig)`); 2ª SIGTERM em <5s pula direto pro hard exit. (b) `subprocess.run(systemctl)` síncrono em `on_daemon_service_restart` migrado para worker thread. (c) `dialog.run()` substituído por `connect("response", destroy)` no diálogo de erro do restart |
| 147 | BUG-BOOT-NOTIFICATION-SPAM-01 | (a) Probe do tray COSMIC: 500ms → 1500ms + 3 retries de 1s, flag persistente em `$XDG_RUNTIME_DIR/.../cosmic_tray_warned.flag` (notifica só uma vez por instalação). (b) `notify_system_warnings` (WirePlumber, udev outdated) default off — opt-in via env `HEFESTO_DUALSENSE4UNIX_SYSTEM_WARNINGS_NOTIFY=1`. (c) `notify_controller_connected/disconnected` com throttle de 30s por chave (elimina rajada em flap espúrio) |

## Resultado

- Release `v3.8.2` no fork (`[REDACTED]/hefesto-dualsense4unix`) com 6 assets: AppImage,
  Flatpak, dois `.deb` (py310/py312), wheel e sdist.
- Validação na máquina: daemon novo subiu sem `OSError [Errno 9]` no `journalctl`, conectou
  ao controle real OU marcou offline em 5s quando hidraw estava bloqueado, e o SIGTERM matou
  GUI em <3s mesmo com mainloop bloqueado por subprocess.
- Daemon zombie pré-existente (que disparou a sessão) ficou imkillable pelo kernel — só morre
  com reboot. Os fixes garantem que **o estado não se repete**, não que o estado atual seja
  desfeito.

## Notas de retrospectiva

- **`fcntl.flock` é por inode, não por path** — apagar o pid file e abrir um novo cria lock
  independente do antigo (o zombie ainda detém o lock no inode antigo, mas o novo inode é
  livre). Foi o que permitiu validar o fix sem rebotar a máquina.
- **Processo em `D (disk sleep)` não morre com SIGKILL nem com cgroup.kill** — Linux não tem
  mecanismo de userspace pra isso. Memória reforçada para sessões futuras: se `pgrep` mostra
  `STAT=D`, abandone tentativas de kill e mire em "como impedir o estado", não em "como sair
  dele".
- **`pkill -f` pode matar o próprio shell** — armadilha já documentada
  (`feedback_pkill_f_mata_proprio_shell`); reforçada quando o pattern incluía
  `hefesto_dualsense4unix.app.main` e atingia o `zsh -c` do harness. Usar `pgrep` + filtro por
  `/proc/<pid>/comm` evita.
- **`ThreadPoolExecutor` com `__exit__` faz `shutdown(wait=True)`** — context manager join-a
  threads vazadas e trava o caller. Para timeout-com-thread-abandonável, usar
  `threading.Thread(daemon=True)` direto, sem executor.
